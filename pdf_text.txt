1.1 Project Description
This project presents the complete design and implementation of a Database
Management System for a Sports Club. The system is designed to manage members,
coaches, staff, sports activities, facilities, equipment, payments, and member
dependents. The database captures all essential relationships between these entities,
enabling eﬃcient data storage, retrieval, and management for the club's daily operations.
1.2 Problem Statement
Managing a sports club involves handling a large volume of interconnected data:
member registrations, enrollment in various sports, coach assignments, facility
bookings, equipment tracking, and payment processing. Without a structured database
system, this data becomes prone to redundancy, inconsistency, and loss. This project
addresses these challenges by designing a normalized relational database that ensures
data integrity, eliminates redundancy, and supports eﬃcient querying for all club
operations.2. Conceptual Design

3.1 Relational Schema (Mapping Rules)
The following mapping rules were applied to convert the ERD to a relational schema:
• Each strong entity maps to its own table with its primary key.
• Weak entity maps to a table with composite PK (owner PK + partial key).
• 1:N relationships: FK placed on the N-side table.
• M:N relationships: new junction (associative) table with composite PK.
• Multivalued attributes: separate table with composite PK.
• Composite attributes: decomposed into atomic columns.
• Derived attributes: NOT stored (computed at query time).

4.1 Data Deﬁnition Language (DDL)
The following SQL statements create all database tables with proper constraints:
-- =============================================
-- Sports Club Database - DDL (CREATE TABLES)
-- =============================================
CREATE DATABASE IF NOT EXISTS sports_club;
USE sports_club;
-- 1. MEMBER
CREATE TABLE MEMBER (
Member_ID INT PRIMARY KEY AUTO_INCREMENT,
Name VARCHAR(100) NOT NULL,
Gender CHAR(1) CHECK (Gender IN ('M','F')),
DOB DATE NOT NULL,
City VARCHAR(50),
Street VARCHAR(100),
ZipCode VARCHAR(10)
);
-- 2. MEMBER_PHONE
CREATE TABLE MEMBER_PHONE (
Member_ID INT,
Phone VARCHAR(20),
PRIMARY KEY (Member_ID, Phone),
FOREIGN KEY (Member_ID) REFERENCES MEMBER(Member_ID) ON DELETE CASCADE
);
-- 3. DEPENDENT
CREATE TABLE DEPENDENT (
Member_ID INT,
Dep_Name VARCHAR(100),
Relationship VARCHAR(50) NOT NULL,
DOB DATE,
PRIMARY KEY (Member_ID, Dep_Name),
FOREIGN KEY (Member_ID) REFERENCES MEMBER(Member_ID) ON DELETE CASCADE
);
-- 4. STAFF
CREATE TABLE STAFF (
Staff_ID INT PRIMARY KEY AUTO_INCREMENT,
Name VARCHAR(100) NOT NULL,
Gender CHAR(1) CHECK (Gender IN ('M','F')),
DOB DATE,
Role VARCHAR(50));
-- 5. FACILITY
CREATE TABLE FACILITY (
Fac_ID INT PRIMARY KEY AUTO_INCREMENT,
Location VARCHAR(100) NOT NULL,
Capacity INT,
Staff_ID INT,
FOREIGN KEY (Staff_ID) REFERENCES STAFF(Staff_ID) ON DELETE SET NULL
);
-- 6. COACH
CREATE TABLE COACH (
Coach_ID INT PRIMARY KEY AUTO_INCREMENT,
Name VARCHAR(100) NOT NULL,
Gender CHAR(1) CHECK (Gender IN ('M','F')),
DOB DATE,
Salary DECIMAL(10,2)
);
-- 7. SPORT
CREATE TABLE SPORT (
Sport_ID INT PRIMARY KEY AUTO_INCREMENT,
Sport_Name VARCHAR(100) NOT NULL,
Fac_ID INT,
Coach_ID INT,
FOREIGN KEY (Fac_ID) REFERENCES FACILITY(Fac_ID) ON DELETE SET NULL,
FOREIGN KEY (Coach_ID) REFERENCES COACH(Coach_ID) ON DELETE SET NULL
);
-- 8. ENROLLS (Junction: Member M:N Sport)
CREATE TABLE ENROLLS (
Member_ID INT,
Sport_ID INT,
Enroll_Date DATE NOT NULL,
PRIMARY KEY (Member_ID, Sport_ID),
FOREIGN KEY (Member_ID) REFERENCES MEMBER(Member_ID) ON DELETE CASCADE,
FOREIGN KEY (Sport_ID) REFERENCES SPORT(Sport_ID) ON DELETE CASCADE
);
-- 9. PAYMENT
CREATE TABLE PAYMENT (
Pay_ID INT PRIMARY KEY AUTO_INCREMENT,
Member_ID INT NOT NULL,
Amount DECIMAL(10,2) NOT NULL,
Pay_Date DATE NOT NULL,
FOREIGN KEY (Member_ID) REFERENCES MEMBER(Member_ID) ON DELETE CASCADE
);
-- 10. EQUIPMENTCREATE TABLE EQUIPMENT (
Equip_ID INT PRIMARY KEY AUTO_INCREMENT,
Equip_Name VARCHAR(100) NOT NULL,
Condition_ VARCHAR(50)
);
-- 11. USES (Junction: Sport M:N Equipment)
CREATE TABLE USES (
Sport_ID INT,
Equip_ID INT,
PRIMARY KEY (Sport_ID, Equip_ID),
FOREIGN KEY (Sport_ID) REFERENCES SPORT(Sport_ID) ON DELETE CASCADE,
FOREIGN KEY (Equip_ID) REFERENCES EQUIPMENT(Equip_ID) ON DELETE CASCADE
);4.2 Data Manipulation Language (DML)

This project successfully demonstrated the complete lifecycle of database design and
implementation for a Sports Club management system. Starting from the conceptual
design phase with an Entity-Relationship Diagram (ERD) using Chen Notation, we
identiﬁed eight core entities — Member, Dependent, Payment, Sport, Coach, Equipment,
Facility, and Staff — along with their attributes and inter-relationships.
The Enhanced ERD (EERD) introduced a generalization hierarchy by abstracting shared
attributes into a PERSON superclass with Total, Overlapping specialization constraints,
reﬂecting the real-world scenario where a person can simultaneously hold multiple roles
(e.g., a coach who is also a club member).
The logical design phase mapped the conceptual model to a relational schema
consisting of 11 normalized tables. All tables were veriﬁed to satisfy First, Second, and
Third Normal Forms (1NF, 2NF, 3NF), ensuring the elimination of data redundancy and
the preservation of data integrity.
The physical implementation translated the schema into executable SQL code (DDL for
table creation, DML for data population), which was successfully deployed on a MySQL
database through phpMyAdmin. A simple web-based GUI was also developed to
facilitate data entry operations.
This project reinforced key database engineering concepts including normalization
theory, ER modeling, schema mapping rules, and the practical bridge between
theoretical design and working software systems.
